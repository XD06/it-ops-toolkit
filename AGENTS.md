# Agent 工作指南

本仓库是一个中小企业 IT 运维工具平台。文档就是项目记忆。任何后续开发都不能只依赖聊天记录或临时理解。

## 开始工作前必须阅读

在修改代码或文档前，先按顺序阅读：

1. `docs/00-project-vision.md`
2. `docs/01-architecture-overview.md`
3. `docs/02-module-map.md`
4. `docs/04-development-rules.md`
5. 如果处理具体模块，阅读 `docs/modules/` 下对应模块文档。

## 项目基本方向

- 平台优先，CLI 优先。
- 第一阶段做实用工具箱，不做大而全系统。
- 架构必须预留 Web Console、API、AI Copilot、Agent Runner。
- 模块必须高内聚、低耦合。
- 入口层不能承载业务逻辑。
- 外部协议、命令、脚本、设备能力必须通过 Adapter 接入。
- 所有执行结果都要尽量结构化，方便报告、Web、AI 和审计复用。
- AI 和 Agent 是受控助手，不是无限权限操作者。

## 开发边界

不要把以下逻辑写进 CLI 或 Web：

- 资产识别规则
- 巡检判断规则
- 故障诊断流程
- 安全风险分类
- 自动化执行逻辑
- 报告生成逻辑

CLI 和 Web 只能调用应用服务或领域服务。

## 文档维护要求

- 新增模块时，先写模块设计文档。
- 改变模块边界时，更新 `docs/02-module-map.md`。
- 做出重要架构选择时，新增 ADR。
- 发现高频术语时，补充 `docs/glossary.md`。
- 如果暂时不做某能力，记录到路线图，而不是隐含在脑子里。
