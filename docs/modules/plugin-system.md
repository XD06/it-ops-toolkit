# 插件与探针系统模块

## 模块职责

插件与探针系统负责定义平台如何接入外部能力。

它是实现高内聚、低耦合的关键。Ping、DNS、TCP、HTTP、SNMP、SSH、WinRM、WMI、PowerShell、Python 脚本都不应该散落在业务模块里，而应该通过 Adapter 或 Probe 接入。

## 现实场景

运维工具经常需要调用不同能力：

- Windows 用 PowerShell 和 WMI。
- Linux 用 SSH。
- 网络设备用 SNMP 或 SSH。
- 服务检查用 TCP 和 HTTP。
- DNS 问题用 nslookup 类能力。

如果每个模块自己随便调用外部命令，后续会很难维护、测试和替换。

## 不负责什么

本模块不负责：

- 定义业务判断。
- 直接生成报告。
- 直接保存资产。
- 允许运行任意不受控脚本。

## 输入

输入可以包括：

- Probe 请求。
- Adapter 配置。
- 目标。
- 超时和重试策略。
- 凭据引用。

## 输出

输出应是标准化 ProbeResult，例如：

- 目标。
- Probe 类型。
- 状态。
- 耗时。
- 观察值。
- 原始错误。
- 归一化错误。
- 证据。

## 依赖模块

依赖：

- 配置中心。
- 日志与观测性。
- 凭据与密钥管理。

被依赖：

- 资产与拓扑。
- 网络与服务巡检。
- 轻量安全与合规。
- 自动化动作。
- 场景化故障诊断。

## 第一阶段范围

第一阶段内置 Probe：

- Ping。
- DNS。
- TCP。
- HTTP。

需要先定义统一接口和统一结果结构。

## 未来扩展

后续可扩展：

- SNMP。
- SSH。
- WinRM。
- WMI。
- PowerShell。
- Python 插件。
- 厂商设备 Adapter。
- 插件目录。
- 插件权限控制。

## 验收标准

第一阶段完成时，应满足：

- 至少有 4 个内置 Probe。
- Probe 输出结构一致。
- 业务模块不直接调用系统命令。
- 单个 Probe 失败不会导致整个任务崩溃。
- 错误信息可诊断。
