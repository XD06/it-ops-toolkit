# 场景化故障诊断模块

## 模块职责

场景化故障诊断模块负责把用户现象转成标准化排障流程。

它不是简单地跑一个命令，而是根据场景组织多个检查，并输出：

- 可能原因。
- 证据。
- 下一步建议。
- 需要人工确认的事项。

## 现实场景

典型用户描述往往不专业：

- “上不了网。”
- “内网系统打不开。”
- “远程桌面连不上。”
- “网络很慢。”
- “打印机不能用。”
- “有些人能打开，有些人不能打开。”

初级运维最容易卡在“不知道第一步查什么”。诊断模块的价值是把经验固化成流程。

## 不负责什么

本模块不负责：

- 实现所有底层探测。
- 自动执行危险修复。
- 替代人工最终判断。
- 发送通知。
- 维护资产信息。

它调用资产、巡检、本机检查和安全检查等模块来形成诊断。

## 输入

输入可以包括：

- 诊断场景。
- 用户所在主机或 IP。
- 目标业务系统。
- 当前网络环境。
- 相关巡检结果。
- 历史资产信息。

## 输出

输出应包括：

- 诊断场景。
- 执行过的检查步骤。
- 每一步结果。
- 可能原因列表。
- 证据引用。
- 建议下一步。
- 风险提示。
- 是否需要升级处理。

## 依赖模块

依赖：

- 资产与拓扑。
- 网络与服务巡检。
- 插件与探针系统。
- 数据存储。
- 报告中心。

被依赖：

- AI 运维助手。
- Agent Runner。
- Web Console。

## 第一阶段范围

第一阶段可以先定义诊断流程格式，选择 1-2 个场景实现。

推荐第一批场景：

- 上不了网。
- 内网系统打不开。

每个场景都要输出检查步骤和证据，而不是只输出一句结论。

## 未来扩展

后续可扩展：

- 远程桌面连不上。
- DNS 异常。
- 网络很慢。
- 打印机不可达。
- VPN 异常。
- 邮件异常。
- 账号锁定。
- Agent 自动执行只读诊断流程。

## 验收标准

第一阶段完成时，应满足：

- 至少有一个完整诊断场景。
- 诊断步骤可配置或可维护。
- 每个结论都有证据来源。
- 输出包含下一步建议。
- 不执行高风险动作。
