# 日志与观测性模块

## 模块职责

日志与观测性模块负责记录平台自身运行情况，帮助开发者和运维人员排查工具本身的问题。

它不是业务巡检模块，而是平台内部可诊断能力。

## 现实场景

工具本身也会出问题：

- 某个 Probe 超时。
- DNS 检查失败但原因不清楚。
- HTTP 请求被代理影响。
- 报告生成失败。
- 配置解析失败。
- AI 调用失败。
- 某次扫描很慢。

没有日志，工具就会变成新的黑箱。

## 不负责什么

本模块不负责：

- 判断业务服务是否健康。
- 保存全部业务结果。
- 替代报告中心。
- 暴露敏感信息。

## 输入

输入可以包括：

- 应用日志。
- 任务日志。
- Adapter 错误。
- 耗时数据。
- 异常堆栈。
- 脱敏后的上下文。

## 输出

输出可以包括：

- 本地日志文件。
- 结构化日志。
- 错误摘要。
- 性能耗时记录。
- 任务日志引用。

## 依赖模块

依赖：

- 配置中心。
- 凭据与密钥管理。

被依赖：

- 全部模块。

## 第一阶段范围

第一阶段建议支持：

- 基础日志级别。
- 任务执行日志。
- Adapter 错误日志。
- 配置解析错误。
- 报告生成错误。
- 敏感信息脱敏。

## 未来扩展

后续可扩展：

- 结构化 JSON 日志。
- Web 日志查看。
- 性能指标。
- 外部日志系统集成。
- 调试包导出。

## 验收标准

第一阶段完成时，应满足：

- 出错时有可读原因。
- 单个 Adapter 失败能定位。
- 日志不泄露敏感信息。
- 日志能关联任务 ID。
