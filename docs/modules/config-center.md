# 配置中心模块

## 模块职责

配置中心负责管理平台运行所需的配置，避免网段、目标、端口、报告路径、超时、重试等信息散落在代码和脚本里。

配置中心是第一阶段必须优先完成的基础模块。

## 现实场景

中小企业运维中常见配置包括：

- 办公网段。
- 服务器地址。
- DNS 服务器。
- 网关。
- 关键业务系统 URL。
- 常用端口。
- 巡检目标。
- 报告输出目录。
- 通知接收人。

如果这些都写死在脚本里，后续维护会非常痛苦。

## 不负责什么

本模块不负责：

- 执行扫描。
- 保存大量历史结果。
- 明文保存敏感密码。
- 提供完整 Web 配置界面。
- 多用户权限控制。

## 输入

输入可以包括：

- 配置文件。
- CLI 参数。
- 环境变量。
- 未来 Web 配置。
- 默认配置模板。

## 输出

输出应包括：

- 解析后的配置对象。
- 配置校验结果。
- 默认值填充结果。
- 错误提示。

## 依赖模块

依赖：

- 日志与观测性。
- 凭据与密钥管理。

被依赖：

- 资产与拓扑。
- 网络与服务巡检。
- 场景化故障诊断。
- 安全合规。
- 自动化动作。
- 报告中心。
- 任务调度。

## 第一阶段范围

第一阶段建议支持本地配置文件。

配置内容包括：

- 网段。
- 目标列表。
- 端口配置。
- 超时配置。
- 重试配置。
- 输出路径。
- 报告格式。

Phase 1 配置格式已选择 YAML，决策记录见 `docs/adr/0004-phase-1-yaml-config.md`。TOML 和 JSON 作为备选方案保留在 ADR 中。

## 未来扩展

后续可扩展：

- Web 配置编辑。
- 配置版本管理。
- 配置校验规则。
- 环境区分。
- 凭据引用。
- 配置导入导出。

## 验收标准

第一阶段完成时，应满足：

- 能读取本地配置。
- 能校验配置错误。
- 能提供清晰错误提示。
- 业务模块通过配置中心读取配置。
- 没有大量硬编码目标。
