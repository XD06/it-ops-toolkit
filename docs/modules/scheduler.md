# 任务调度中心模块

## 模块职责

任务调度中心负责管理任务执行。

第一阶段可以先不做真正的定时调度，但必须有任务记录概念。否则后续无法做历史、报告、Web Dashboard、通知和审计。

## 现实场景

运维任务包括：

- 手动运行一次资产扫描。
- 手动运行一次巡检。
- 每天早上自动巡检。
- 每周生成一次报告。
- 故障时运行诊断流程。
- 未来 Agent 运行一个工作流。

这些都应该被记录成 TaskRun。

## 不负责什么

本模块不负责：

- 具体业务检查。
- 生成报告内容。
- 发送通知。
- 保存全部探测细节。

## 输入

输入可以包括：

- 任务类型。
- 目标。
- 参数。
- 调度策略。
- 超时。
- 重试。
- 执行人。

## 输出

输出应包括：

- 任务 ID。
- 任务状态。
- 开始时间。
- 结束时间。
- 结果引用。
- 错误摘要。
- 日志引用。

## 依赖模块

依赖：

- 数据存储。
- 权限与审计。
- 日志与观测性。

被依赖：

- CLI。
- Web Console。
- Agent Runner。
- 通知中心。

## 第一阶段范围

第一阶段建议支持：

- 手动任务记录。
- 任务状态：pending、running、success、failed、cancelled。
- 任务开始和结束时间。
- 结果引用。
- 错误摘要。

不急着做复杂 cron 调度。

## 未来扩展

后续可扩展：

- 定时任务。
- 任务队列。
- 并发控制。
- 失败重试。
- 任务取消。
- Web 任务管理。
- Worker 模式。

## 验收标准

第一阶段完成时，应满足：

- 每次 CLI 执行能形成任务记录。
- 任务能关联结果。
- 失败能记录原因。
- 后续 Web 可以读取任务历史。
