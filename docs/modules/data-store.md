# 数据存储模块

## 模块职责

数据存储模块负责保存平台结构化数据。

它不是业务规则模块，只提供数据读写能力和稳定的数据模型。

## 现实场景

如果不保存历史，系统只能回答“现在是什么情况”，不能回答：

- 这个设备什么时候第一次出现。
- 这个 IP 最近是否换过设备。
- 某服务从什么时候开始异常。
- 上周巡检和今天有什么变化。
- 哪次任务生成了这份报告。
- AI 总结基于哪些结果。

所以数据存储是 Web、AI、报告和审计的基础。

## 不负责什么

本模块不负责：

- 判断风险。
- 执行探测。
- 生成报告内容。
- 发送通知。
- 保存明文敏感凭据。

## 输入

输入可以包括：

- 资产记录。
- ProbeResult。
- TaskRun。
- Finding。
- Report。
- Event。
- KnowledgeEntry。

## 输出

输出包括：

- 查询结果。
- 保存结果。
- 数据引用 ID。
- 错误信息。

## 依赖模块

依赖：

- 配置中心。
- 日志与观测性。

被依赖：

- 几乎所有业务模块。
- Web Console。
- AI 运维助手。
- 报告中心。
- 权限与审计。

## 第一阶段范围

第一阶段可以使用本地文件或轻量数据库。

需要保存：

- 资产。
- 任务记录。
- 探测结果。
- 报告元数据。

具体存储技术在实现计划中决定。

## Phase 6 扩展：历史趋势查询与聚合

### 新增能力

- **按时间范围查询**：`list_probe_results_between(start, end)` 查询指定时间范围内的探测结果。
- **按探针类型和目标筛选**：在时间范围查询基础上，支持 `probe_type` 和 `target` 筛选。
- **聚合统计**：`get_probe_stats(probe_type, target, start, end, granularity)` 按天/小时分组，计算 avg/min/max/p95。
- **状态分布**：`get_status_distribution(probe_type, target, start, end)` 统计成功/失败/超时分布。

### 聚合粒度

支持两种聚合粒度：

- `hourly`：按小时分组，适合短时间范围（24 小时内）。
- `daily`：按天分组，适合长时间范围（7 天以上）。

### 统计指标

对于数值型指标（如 RTT、丢包率），计算：

- `count`：样本数。
- `avg`：平均值。
- `min`：最小值。
- `max`：最大值。
- `p95`：95 分位数。

对于状态型指标，计算：

- `success_count`：成功次数。
- `failed_count`：失败次数。
- `timeout_count`：超时次数。
- `success_rate`：成功率百分比。

### 设计原则

- 聚合在 SQLite 层完成，减少 Python 内存消耗。
- 趋势数据是只读的，不修改原始数据。
- 聚合结果结构化，可被 CLI、Web、AI 复用。
- 趋势查询不依赖外部时间序列数据库。

## 验收标准

第一阶段完成时，应满足：

- 能保存结构化结果。
- 能按任务查询结果。
- 能按资产查询历史。
- 能被报告模块读取。
- 存储接口不绑定某个 UI。

Phase 6 完成时，应额外满足：

- 能查询指定时间范围内的探测结果。
- 能按天/小时生成聚合统计。
- 聚合结果包含 avg/min/max/p95。
- 能统计状态分布。
- 趋势查询可被 CLI、Web 和 AI 复用。
