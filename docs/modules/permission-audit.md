# 权限与审计模块

## 模块职责

权限与审计模块负责记录谁在什么时候对什么目标执行了什么动作，以及结果如何。

第一阶段即使没有多用户登录，也应该保留审计结构。因为未来 Web、Agent、自动化动作都依赖它。

## 现实场景

常见问题：

- 不知道某次扫描是谁跑的。
- 不知道某份报告来自哪次任务。
- 自动化动作执行失败后没有记录。
- 高风险操作没有审批痕迹。
- 多人使用工具后无法追踪责任。

审计模块的价值是让操作可追溯。

## 不负责什么

本模块不负责：

- 实现完整身份认证系统。
- 保存明文密码。
- 判断业务故障原因。
- 执行自动化动作。

## 输入

输入可以包括：

- 执行人。
- 动作类型。
- 目标。
- 风险等级。
- 时间。
- 参数摘要。
- 审批信息。
- 执行结果。

## 输出

输出包括：

- 审计事件 ID。
- 审计记录。
- 查询结果。
- 权限检查结果。

## 依赖模块

依赖：

- 数据存储。
- 配置中心。
- 日志与观测性。

被依赖：

- 自动化动作。
- Agent Runner。
- CLI。
- Web Console。
- 任务调度中心。

## 第一阶段范围

第一阶段建议支持：

- 记录命令执行。
- 记录任务 ID。
- 记录目标摘要。
- 记录风险等级。
- 记录成功或失败。

不急着实现复杂 RBAC。

## 未来扩展

后续可扩展：

- 用户登录。
- 角色权限。
- 审批流程。
- 操作回放。
- 审计报表。
- 高风险动作拦截。

## 验收标准

第一阶段完成时，应满足：

- 任务执行有审计记录。
- 自动化动作有风险等级。
- 审计记录能关联任务和结果。
- 不记录明文敏感信息。
