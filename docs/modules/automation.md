# 自动化动作模块

## 模块职责

自动化动作模块负责把常见运维动作标准化，并记录执行过程。

它的重点不是“能自动做多少事”，而是“哪些动作可以安全地自动做，哪些必须审批”。

## 现实场景

运维中经常有重复动作：

- 清 DNS 缓存。
- 导出诊断信息。
- 批量测试端口。
- 批量生成连通性矩阵。
- 检查证书过期时间。
- 收集 Windows 本机信息。
- 未来备份网络设备配置。

如果这些动作没有分级和审计，就容易从“方便工具”变成“误操作工具”。

## 不负责什么

本模块不负责：

- 自己判断业务故障原因。
- 绕过审批执行高风险操作。
- 保存资产主数据。
- 发送通知。

## 输入

输入可以包括：

- 动作类型。
- 目标。
- 参数。
- 风险等级。
- 执行人。
- 审批状态。
- 超时和重试策略。

## 输出

输出应包括：

- 动作名称。
- 目标。
- 开始时间。
- 结束时间。
- 状态。
- 输出结果。
- 错误信息。
- 风险等级。
- 审计记录引用。

## 依赖模块

依赖：

- 配置中心。
- 插件与探针系统。
- 权限与审计。
- 数据存储。
- 日志与观测性。

被依赖：

- Agent Runner。
- Web Console。
- CLI。
- 报告中心。

## 第一阶段范围

第一阶段只支持只读动作和极少数低风险本机动作。

推荐优先：

- 导出诊断包。
- 批量端口测试。
- 批量连通性测试。
- 证书到期检查。

不建议第一阶段做：

- 重启远程服务。
- 修改网络设备配置。
- 批量关闭端口。
- 禁用账号。

## 未来扩展

后续可扩展：

- 审批流程。
- 回滚信息。
- 远程设备配置备份。
- Runbook 工作流。
- Agent 自动执行低风险步骤。
- 高风险动作人工确认。

## 验收标准

第一阶段完成时，应满足：

- 每个动作都有风险等级。
- 每次执行都有记录。
- 失败能记录原因。
- 高风险动作不会被默认执行。
- 动作逻辑不写在 CLI 内部。
