# 通知中心模块

## 模块职责

通知中心负责把需要提醒的信息发送到外部渠道。

它不判断什么是异常，只负责把其他模块产生的通知事件发送出去。

## 现实场景

中小企业运维常见通知需求：

- 每日巡检完成后发送摘要。
- 关键服务不可达时提醒值班人员。
- 发现未知设备时提醒。
- 证书快过期时提醒。
- 自动化任务失败时提醒。

常见渠道：

- 邮件。
- 企业微信。
- 钉钉。
- 飞书。
- Webhook。

## 不负责什么

本模块不负责：

- 执行巡检。
- 判断风险等级。
- 生成报告内容。
- 保存主要历史结果。
- 管理用户权限。

## 输入

输入可以包括：

- 通知事件。
- 目标渠道。
- 收件人。
- 标题。
- 正文。
- 严重级别。
- 报告链接或文件路径。

## 输出

输出应包括：

- 发送状态。
- 渠道。
- 发送时间。
- 错误信息。
- 重试次数。

## 依赖模块

依赖：

- 配置中心。
- 数据存储。
- 日志与观测性。

被依赖：

- 报告中心。
- 网络与服务巡检。
- 轻量安全与合规。
- 任务调度中心。

## 第一阶段范围

第一阶段可以只做接口设计，不一定实现真实通知渠道。

如果要实现，建议先做最简单的：

- 邮件。
- Webhook。

## 未来扩展

后续可扩展：

- 企业微信。
- 钉钉。
- 飞书。
- 通知模板。
- 通知降噪。
- 通知升级。
- 值班表。

## 验收标准

第一阶段完成时，应满足：

- 通知事件结构已定义。
- 其他模块不直接调用具体通知渠道。
- 未来能新增渠道而不改业务模块。
