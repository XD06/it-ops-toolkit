# 风险策略

## 文档目的

这份文档定义自动化、AI 和 Agent 相关的风险边界。

项目可以追求效率，但不能让工具变成误操作放大器。

## 风险等级

### read_only

只读操作。

示例：

- Ping。
- DNS 查询。
- TCP 端口检查。
- HTTP 状态检查。
- 资产扫描。
- 读取本机信息。
- 生成报告。

默认允许。

### low_change

低风险变更。

示例：

- 清理本机 DNS 缓存。
- 生成诊断包。
- 刷新本地状态。

可以在后续阶段支持，但必须记录。

### high_change

高风险变更。

示例：

- 重启服务。
- 修改防火墙。
- 修改网络设备配置。
- 批量远程执行命令。
- 禁用账号。
- 删除文件或配置。
- 封禁设备。

第一阶段禁止自动执行。

## 默认策略

第一阶段默认策略：

- 默认只执行 read_only。
- low_change 需要明确命令。
- high_change 不实现自动执行。
- 所有动作都记录 TaskRun。
- 所有失败都记录错误。

## dry-run

未来所有 low_change 和 high_change 动作都应支持 dry-run。

dry-run 应输出：

- 将要执行的动作。
- 目标。
- 风险等级。
- 预期影响。
- 是否需要审批。

## 审批要求

未来 high_change 必须满足：

- 明确执行人。
- 明确目标。
- 明确动作。
- 明确风险。
- 明确审批人。
- 明确回滚或恢复建议。

## AI 风险边界

AI 可以：

- 总结。
- 解释。
- 建议。
- 生成报告。
- 匹配 SOP。

AI 不应默认：

- 直接执行 high_change。
- 隐藏事实来源。
- 把推断说成确定事实。
- 输出敏感信息。
- 绕过人工确认。

## Agent 风险边界

Agent 可以先从只读工作流开始。

Agent 必须：

- 调用预定义工具。
- 记录每一步。
- 遇到 high_change 停止并请求审批。
- 遇到不确定情况停止并说明原因。
- 允许人工接管。

Agent 不允许：

- 自己发明未授权命令。
- 绕过审批。
- 执行没有风险等级的动作。
- 隐藏失败步骤。

## 敏感信息规则

日志、报告、AI 输入和 Agent 上下文不得包含：

- 明文密码。
- Token。
- 私钥。
- 未脱敏连接串。
- 不必要的个人信息。

## 第一阶段验收标准

第一阶段应满足：

- 自动化动作有风险等级。
- CLI 默认只执行 read_only。
- 文档明确 high_change 不自动执行。
- 日志和报告不输出敏感信息。
- AI 和 Agent 不进入实现范围，但边界已定义。
