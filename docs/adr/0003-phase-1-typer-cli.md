# ADR 0003：Phase 1 选择 Typer 作为 CLI 框架

## 状态

已接受，作为 Phase 1 CLI 方向。

## 背景

第一阶段 CLI 是主要用户入口。CLI 需要：

- 命令分组清楚。
- 参数提示清楚。
- 错误信息友好。
- 方便后续增加命令。
- 不把业务逻辑写进命令函数。

## 决策

Phase 1 使用 Typer 作为 CLI 框架。

Typer 官方文档：[Typer](https://typer.tiangolo.com/)。

CLI 输出可以配合 Rich 做表格、颜色和进度显示。Rich 官方文档：[Rich](https://rich.readthedocs.io/)。

## 备选方案

### argparse

优点：

- Python 标准库。
- 无第三方依赖。
- 稳定。

缺点：

- 命令结构和类型提示体验不如 Typer。
- 对复杂命令组的维护成本更高。

### Click

优点：

- 成熟稳定。
- Typer 底层基于 Click。

缺点：

- 类型提示和现代 Python 开发体验不如 Typer 直接。

## 后果

正面影响：

- 命令定义清晰。
- 类型提示友好。
- 帮助信息自动生成。
- 后续扩展命令较方便。

负面影响：

- 增加第三方依赖。
- 打包发布时要处理依赖。
- 如果未来需要极简标准库实现，可能需要迁移到 argparse。

## 执行要求

- CLI 命令函数只做参数解析和调用应用服务。
- 业务逻辑不能写在 CLI 命令函数中。
- 所有命令必须支持清晰错误输出。
- 重要命令应支持 `--json`，方便脚本调用。
