# ADR 0002：Phase 1 选择 Python 作为主要实现语言

## 状态

已接受，作为 Phase 1 技术方向。

## 背景

本项目第一阶段要做 CLI 运维工具箱，主要能力包括：

- 读取配置。
- 执行网络探测。
- 调用系统命令或脚本。
- 保存本地结果。
- 生成报告。
- 未来接入 AI。

目标用户是桌面运维、初级网络工程师和中小企业 IT 人员。项目需要容易理解、容易扩展、适合脚本化，也要能在 Windows 环境工作。

## 决策

Phase 1 选择 Python 作为主要实现语言。

建议版本：

- Python 3.12 或更新版本。
- 编码时避免使用过新的语法特性，方便后续部署。

Python 官方文档入口：[Python Documentation](https://docs.python.org/3/)。

## 备选方案

### Go

优点：

- 单文件二进制发布方便。
- 并发和网络能力强。
- 部署依赖少。

缺点：

- 对初级运维人员不如 Python 友好。
- 和 PowerShell、AI、数据处理脚本协作时不如 Python 灵活。
- 前期迭代速度略慢。

### PowerShell

优点：

- Windows 环境天然可用。
- 适合桌面运维和 Windows 信息采集。

缺点：

- 跨平台和长期模块化不如 Python。
- Web、AI、数据模型和测试体系会受限制。
- 复杂项目容易演变成脚本集合。

### Node.js

优点：

- Web 生态强。
- 未来做前端和 API 方便。

缺点：

- 桌面运维和系统探测场景不如 Python 自然。
- Windows 本机运维脚本生态不如 PowerShell/Python 组合。

## 后果

正面影响：

- 开发快。
- 运维人员容易理解。
- AI、报告、数据处理生态成熟。
- 可以通过 Adapter 调用 PowerShell。
- 未来 Web API 可以使用 Python Web 框架承接。

负面影响：

- 分发到没有 Python 的机器时需要解决打包问题。
- 依赖管理需要规范。
- 长期如果需要单文件部署，可能需要 PyInstaller、uv、pipx 或重新评估 Go。

## 执行要求

- 核心逻辑写成模块，不写成零散脚本。
- PowerShell 只能作为 Adapter 能力被调用，不作为主架构。
- 避免业务逻辑散落在 CLI 命令函数里。
- 第一阶段要尽早建立测试入口。
