# ADR 0004：Phase 1 选择 YAML 作为配置格式

## 状态

已接受，作为 Phase 1 配置方向。

## 背景

第一阶段需要让运维人员配置：

- 网段。
- 巡检目标。
- 端口。
- 超时。
- 重试。
- 报告路径。
- 本地存储路径。

配置文件需要人工可读、可注释、可复制修改。

## 决策

Phase 1 使用 YAML 作为配置文件格式。

YAML 规范入口：[YAML Specification](https://yaml.org/spec/)。

配置结构见 [配置文件设计](../config-schema.md)。

配置校验建议使用 Pydantic 或等价模型校验工具。Pydantic 官方文档：[Pydantic](https://docs.pydantic.dev/)。

## 备选方案

### JSON

优点：

- 标准化强。
- 机器解析简单。

缺点：

- 不适合大量人工编辑。
- 不支持注释。
- 对运维配置不够友好。

### TOML

优点：

- 人工可读。
- Python 标准库支持 `tomllib` 读取。

缺点：

- 表达复杂嵌套结构不如 YAML 直观。
- 对列表型巡检目标配置稍显笨重。

### INI

优点：

- 简单。
- Windows 用户熟悉。

缺点：

- 层级结构弱。
- 不适合复杂扫描配置。

## 后果

正面影响：

- 配置可读性强。
- 支持注释。
- 适合网段、目标、端口、报告等层级配置。

负面影响：

- 需要第三方 YAML 解析库。
- YAML 缩进错误可能导致配置问题。
- 必须做严格校验和清晰错误提示。

## 执行要求

- 必须提供 `ops config init` 生成示例配置。
- 必须提供 `ops config validate` 校验配置。
- 配置错误要指出字段路径。
- 不鼓励在 YAML 中保存明文凭据。
