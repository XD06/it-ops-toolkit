# ADR 0001：平台优先，CLI 优先

## 状态

已接受，作为第一版架构方向。

## 背景

项目长期目标是做一个 IT 运维工具平台，而不是一堆零散脚本。

但如果第一阶段就同时实现 CLI、Web、API、AI、Agent、权限、审计、通知、调度和所有业务模块，复杂度会过高，很可能长期没有可用成果。

反过来，如果只从临时脚本开始，后面想接 Web、AI 和 Agent 时，又很容易推翻重构。

因此需要同时满足两个目标：

- 架构上不能短视。
- 交付上不能贪大。

## 决策

采用“平台优先，CLI 优先”的路线。

平台优先表示：

- 第一版就设计模块边界。
- 第一版就定义结构化结果。
- 第一版就把外部能力封装成 Adapter。
- 第一版就考虑任务记录、报告、审计和未来 Web/AI/Agent。

CLI 优先表示：

- 第一阶段先交付命令行工具。
- CLI 是入口层，不是业务逻辑中心。
- 领域服务要能被未来 Web 和 API 复用。

## 后果

正面影响：

- 第一阶段能尽快做出实用工具。
- 后续 Web 可以复用已有服务。
- AI 可以读取结构化结果。
- Agent 未来有清晰边界。
- 项目不容易变成脚本堆。

负面影响：

- 第一阶段需要更多设计纪律。
- 有些接口会先于完整消费者出现。
- 需要持续维护文档，否则架构约束会失效。

## 执行要求

- CLI 不直接调用底层命令。
- 业务判断写在领域服务。
- 外部协议和脚本写成 Adapter。
- 探测结果必须结构化。
- 报告从结构化结果生成。
- AI 不直接拥有执行权限。
- 高风险动作必须预留审批和审计。
