# 项目愿景

## 项目名称

暂定名称：`IT Ops Toolkit Platform`。

中文可以理解为：中小企业 IT 运维工具平台。

## 一句话目标

为桌面运维、IT 网关、值班人员和初级网络工程师提供一个从 CLI 起步、未来可扩展到 Web Console、AI Copilot 和受控 Agent 的模块化运维平台。

## 背景

中小企业的 IT 运维工作往往不是单一岗位能清晰覆盖的。一个人可能上午处理员工电脑无法联网，下午排查打印机，晚上看业务系统端口是否异常，第二天还要整理巡检报告。

常见现实情况包括：

- 网络设备、Windows 电脑、服务器、打印机、摄像头、NAS、AP、防火墙、云服务混在一起。
- 资产信息不完整，IP、MAC、用途、负责人、开放端口经常靠 Excel 或个人记忆维护。
- 故障排查依赖经验，初级人员面对“上不了网”“系统打不开”“远程连不上”时容易没有清晰步骤。
- 巡检和报告经常靠手工，重复劳动多。
- 安全基础工作容易被忽略，比如未知设备、暴露端口、补丁滞后、备份不清晰。
- AI 已经能辅助总结、解释和生成报告，但如果没有结构化数据，AI 只能看零散文本，效果不稳定。

本项目希望把这些场景整理成可复用、可扩展、可审计的工具平台。

## 长期愿景

长期系统应包含以下入口：

- **CLI**：现场快速运行命令，适合桌面支持、网络排障、批量检查。
- **Web Console**：查看资产、任务、巡检状态、报告、告警、配置。
- **API**：为其他脚本、系统、自动化平台提供调用入口。
- **AI Copilot**：解释巡检结果、总结故障、生成排障建议、编写日报周报。
- **Agent Runner**：在审批、权限、审计和风险控制下执行预定义运维流程。

长期系统应包含以下能力：

- 资产发现与基础拓扑。
- 网络和服务健康巡检。
- 场景化故障诊断。
- 轻量安全与合规检查。
- 常见运维动作自动化。
- 报告生成与故障复盘。
- 通知与告警。
- 知识库和 SOP 沉淀。
- AI 总结和建议。
- 受控 Agent 执行。

## 第一阶段策略

第一阶段不追求一次做完所有能力。第一阶段只做一个最小但完整的闭环：

```text
配置 -> CLI 命令 -> 探测 Adapter -> 领域服务 -> 结构化结果 -> 报告
```

第一阶段优先做：

- 项目骨架。
- 配置中心。
- CLI 入口。
- Ping、DNS、TCP、HTTP 基础 Adapter。
- 资产发现最小流程。
- 网络与服务巡检最小流程。
- 结构化结果模型。
- 报告输出。
- 本地存储。

这条闭环跑通后，后面的 Web、AI、Agent 都可以复用已有能力，而不是重写一套系统。

## 目标用户

### 桌面运维

需要快速判断用户电脑问题，例如：

- 是否拿到正确 IP。
- 网关是否可达。
- DNS 是否正常。
- 内网系统端口是否通。
- 远程桌面是否可连接。
- 是否能一键生成诊断包。

### 初级网络工程师

需要批量了解网络状态，例如：

- 某网段哪些 IP 正在使用。
- 新增了哪些未知设备。
- 哪些端口开放。
- 网关、DNS、业务系统是否稳定。
- 网络慢时应该先看哪里。

### IT 网关和值班人员

需要持续了解环境状态，例如：

- 每天巡检是否正常。
- 哪些服务异常。
- 是否有未知设备接入。
- 是否有高风险端口暴露。
- 是否能快速生成日报、周报、故障报告。

### 小型组织管理员

可能不是专职网络工程师，但需要有工具辅助：

- 看懂网络和设备状态。
- 有标准化排查步骤。
- 有简单报告可以交付。
- 有安全基础检查。

## 产品原则

- 实用优先，不追求炫技。
- CLI 先落地，Web 后接入。
- 平台化设计，模块化开发。
- 只读检查优先，危险操作后置。
- 自动化动作要可审计。
- AI 先解释和建议，再考虑执行。
- Agent 必须受控，不能默认拥有无限权限。
- 文档是项目记忆，不能靠聊天记录记需求。
