# 开发规则

## 文档目的

这份文档是项目开发约束。它比“编码风格”更重要，主要用于防止架构失控。

任何功能开发前，都应该确认没有违反这里的规则。

## 架构规则

1. CLI 和 Web 是入口层，不能包含核心业务逻辑。
2. CLI 和 Web 应该调用应用服务或领域服务。
3. 领域服务不能直接依赖具体 UI、数据库实现或外部命令。
4. 外部协议、命令、脚本、设备能力必须封装成 Adapter。
5. 所有探测、巡检、诊断、自动化动作都应产生结构化结果。
6. 报告模块消费结构化结果，不直接执行探测。
7. AI 模块消费结构化结果、报告和知识库，不应成为隐藏业务规则中心。
8. 高风险自动化动作必须预留审批和审计。
9. 任务执行必须有记录。
10. 每个模块都应该能单独测试。

## 模块耦合规则

允许：

- 模块依赖共享模型。
- 模块依赖接口。
- 模块通过应用服务或领域服务协作。

不允许：

- 一个模块直接读取另一个模块的内部数据结构。
- Adapter 反向调用 CLI 或 Web。
- Storage 层写业务判断。
- 把诊断规则只写在 Prompt 里。
- 把外部命令输出当成长期数据结构。

## 风险分级规则

所有动作都要区分风险等级。

### 只读操作

示例：

- Ping。
- DNS 查询。
- TCP 端口检查。
- HTTP 状态检查。
- 资产扫描。
- 读取本机信息。
- 生成报告。

第一阶段优先支持。

### 低风险变更

示例：

- 清理本机 DNS 缓存。
- 生成诊断包。
- 刷新本地网络状态。

可以在后续阶段谨慎支持。

### 高风险变更

示例：

- 重启服务。
- 修改防火墙策略。
- 修改网络设备配置。
- 批量远程执行命令。
- 禁用账号。
- 删除配置或数据。

第一阶段不自动执行。未来必须有权限、审批、审计和回滚方案。

## 数据规则

结构化结果至少应包含：

- 目标。
- 时间。
- 执行方式。
- 状态。
- 原始观察值。
- 归一化判断。
- 错误信息。
- 证据。

禁止只保存人类可读文本而不保存结构化数据。

## AI 规则

AI 可以做：

- 总结。
- 解释。
- 报告草稿。
- 排障建议。
- SOP 匹配。

AI 不能默认做：

- 直接执行高风险动作。
- 绕过权限。
- 隐藏事实来源。
- 把推断伪装成确定结论。
- 输出敏感信息。

AI 输出应尽量区分：

- 已确认事实。
- 基于证据的推断。
- 建议操作。
- 需要人工确认的事项。

## 文档规则

- 新增模块前，先写模块设计文档。
- 改变模块边界时，更新模块地图。
- 做出重要架构选择时，新增 ADR。
- 新增高频术语时，更新术语表。
- 延后实现的功能写进路线图。
- 不要把“以后再说”的决定留在聊天记录里。
